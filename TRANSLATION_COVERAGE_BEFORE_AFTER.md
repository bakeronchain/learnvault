## CI/CD Validation

### Added Protection

```yaml
- name: i18n key parity check
  run: npm run i18n-check
```

This automatic check ensures:

- ✅ No new keys can be added to en.json without updating all locales
- ✅ All supported locales have complete key parity
- ✅ No rogue locale files (like ps.json) are accidentally enabled
- ✅ Build fails with clear error message if translations are incomplete

### Example Output

```
✅ es.json - all keys match en.json
✅ fr.json - all keys match en.json
✅ sw.json - all keys match en.json
✅ All i18n locales have complete key parity
```

## Historical Context

The project originally had this confusing state:

- Spanish translations existed but language wasn't marked as supported
- French had only 3 nav keys (why?)
- Swahili had only 4 scattered keys
- Pashto (ps.json) was an empty pseudo-locale file mixed in

This suggested:

- Partial/abandoned previous work
- Unclear i18n strategy
- Dead code (ps.json)
- User-facing broken experience (language switcher shows languages that don't
  work)

Now it's clear and professional:

- All 4 supported languages are complete
- CI prevents regression
- Documentation explains the strategy
- Dead code removed
